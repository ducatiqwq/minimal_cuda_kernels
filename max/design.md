## Kernel Design Choices

Design choice of `reduce_max_kernel`: use different APIs for different granularities (similar to `kernels/fwht`).
- **Thread-level**: Read the thread-local values (with stride `blockDim.x * gridDim.x`) using `float4` dtype. Compute the thread-level maximum over all these values.
- **Warp-level**:
  - Call `__syncwarp()` to ensure correctness.
  - Use `__reduce_max_sync` to max-reduce the thread-level maximums.
- **Block-level**:
  - Define an array in smem of size `blockDim.x` (a template argument), and write the $i$-th warp-level of `smem[i]`.
  - Call `__syncthreads()`.
  - Since the maximum warps per block is only $32$ for A100/H100, it sufficies to let the $0$-th warp in the thread to perform a single `__reduce_max_sync`. 
- **Grid-level**:
  - Define (allocated *before* launching the kernel) an array in GMEM of size `gridDim.x`.
  - For the $i$-th block, write its block-level maximum to `gmem[i]`.
  - There is not any expensive grid-level synchronization; instead, after the kernel terminates, the **host** manually computes the overall maximum, which should be ultra-fast.
