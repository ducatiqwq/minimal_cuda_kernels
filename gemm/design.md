## Kernel Design Choices

Crucially, note that the optimal GEMM implementation varies a lot between different GPU architectures (especially between Ampere and Hopper GPUs, since the latter introduces TMA, warpgroups, clusters, etc.). Since the author currently only has access to Ampere GPUs, the implementation **is only expected to work for Ampere GPUs**. For Hopper GPUs like H20, the performance might be extremely slow compared to that of `PyTorch` (or `cuBLAS`).

Design choice of `gemm_naive_kernel`: use three granularities.
- Block-level: Each thread block handles `BLOCK_M` many rows from `A` and `BLOCK_N` many rows from `B`.
  - Shared memory (pipelining) logic: load `SMEM_K` columns from `A` or `B` at a time, with `STAGES_K` stages. Use mbarriers to cooperate with computations.
- Warp-level:
  - For each `MMA_M * MMA_N * MMA_K` sub-partition of the `WARP_M * WARP_N * SMEM_K` partition, call `mma.sync` (the PTX instruction corresponding to `MMA_M, MMA_N, MMA_K`).

We use `cutlass` to make the code clearer.

## Other files

These files are generated by LLM.

- `main.cu`: for kernel launching and correctness checking against a CPU reference.
- `Makefile`: for testing/debugging/profiling.
- `main.py`: the baseline for GEMM (implemented natively with `PyTorch` `torch.matmul`).
- `sweep.py`: sweeping for the best hyperparameters (written to `sweep_results.csv` and `sweep_results.json`) 

## Default configuration

- `M = 8192`, `N = 8192`, `K = 8192`
- `A`: `(M, K)`, `B`: `(N, K)`, `C`: `(M, N)`, `dtype`: `float16`
- Total memory: ~192 MB for `A`, `B`, and `C` combined

## Assumptions on Hyperparameters

- `M % BLOCK_M == 0, N % BLOCK_N == 0, K % SMEM_K == 0`
- Number of blocks: `(M / BLOCK_M) * (N / BLOCK_N)`, which must not exceed the hardware limit
- Number of warps per block: `(BLOCK_M / WARP_M) * (BLOCK_N / WARP_N)`, which must not exceed the hardware limit

## Results (on RTX4090D)

- Pytorch kernel: 8.412ms (130.70 TFLOPS)
- Custom kernel: 7.987ms (137.66 TFLOPS)
